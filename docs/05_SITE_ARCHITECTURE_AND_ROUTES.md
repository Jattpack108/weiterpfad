# 05 — SITE ARCHITECTURE AND ROUTES

## Required routes
- `/`
- `/about`
- `/product`
- `/providers`
- `/for-families`
- `/contact`

## Shared navigation logic
The navigation should be simple and stable.
It should not try to expose everything.
Recommended primary items:
- Product
- For families
- Providers
- About
- Contact

A strong call to action can be:
- Early access
- Talk to us
- Join the first wave

## Route intent

### `/`
Main job:
- explain the product fast
- demonstrate taste
- show the matching idea
- build immediate trust

### `/about`
Main job:
- explain the fragmentation problem
- explain why this product exists
- make the strategic gap legible

Possible sections:
- why the search is fragmented
- why more information alone is not enough
- why the transition after school needs its own product logic

### `/product`
Main job:
- explain how Weiterpfad works
- show the staged product architecture

Possible sections:
- profile and matching
- filtering and relevance
- comparison and shortlist later
- provider verification later
- availability and updates later

### `/providers`
Main job:
- explain why providers matter here
- show a credible low-friction path into the system

Possible sections:
- better visibility through better structure
- staged profile maturity
- future update and availability layer
- why this is not an extraction platform

### `/for-families`
Main job:
- be the most human and practical page
- translate the product into the family journey

Possible sections:
- when the search becomes urgent
- what makes it hard
- how Weiterpfad reduces overload
- what families can do with it now
- what will be added later

### `/contact`
Main job:
- capture early interest without pressure
- support family, provider and partner enquiries

Possible sections:
- short intro
- polished form UI
- who this is for
- simple trust note

## Footer logic
The footer should stay minimal and premium.
