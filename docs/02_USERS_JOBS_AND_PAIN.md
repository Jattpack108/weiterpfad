# 02 — USERS, JOBS AND PAIN

## Primary user
A parent or family member trying to understand the next realistic path after school for a young person with disabilities.

## Core job to be done
Help me understand what options exist, which ones fit, and what I should realistically look at next.

## User pain points
- too many scattered sources
- unclear terms and categories
- weak comparability across options
- no obvious starting point
- too much noise for a sensitive decision
- uncertainty around what matters now
- uncertainty around practical fit
- uncertainty around later steps

## What a good product experience should reduce
- tab chaos
- guesswork
- jargon overload
- list overload
- fear of missing a better option
- lack of structure

## User needs the product should satisfy
- a clear entry point
- a limited number of meaningful categories
- a visible filtering logic
- understandable provider information
- practical cues like distance, profile fit and update state
- a sense of progression

## User language implications
The site should avoid:
- technical bureaucracy
- startup hype
- therapeutic grandstanding

The site should use:
- clear labels
- low-noise explanations
- grounded wording
- restrained confidence

## Secondary user: provider
A provider wants to be represented fairly and clearly without doing heavy technical integration work immediately.

## Secondary user jobs
- show who the offer is for
- show what kind of support exists
- be discoverable in a better way
- later keep profile or availability information more current

## Early trust principle
Families should feel:
- this site understands the problem
- this site reduces complexity
- this site is not trying to manipulate me
