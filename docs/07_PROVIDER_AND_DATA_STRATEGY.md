# 07 — PROVIDER AND DATA STRATEGY

## Strategic principle
Start with a structured reading layer.
Then move toward a provider network layer.
Then move toward update and availability usefulness.

## Layer 1 — Structured source layer
Purpose:
- prove category structure
- prove matching logic
- make the site useful early

What it means:
- normalised local mock data in the codebase
- provider / offer structure that can later absorb real sources
- no claim of complete live data

## Layer 2 — Verified provider profile layer
Purpose:
- improve trust
- improve accuracy
- reduce ambiguity in how providers appear

What it means:
- providers can later verify or claim profiles
- profiles are structured, not free-form chaos
- “last updated” logic can become meaningful later

## Layer 3 — Update and availability layer
Purpose:
- turn discovery into operational usefulness

What it means:
- availability states
- waitlist signals
- change indicators
- later enquiry / response flows

## Offer object fields to normalise
- id
- name
- provider
- category
- city
- region
- distance
- support tags
- transition relevance
- short description
- practical note
- updated state
- availability state later
- next step later

## Why this staged strategy matters
It allows Weiterpfad to launch with clarity and credibility without pretending the hardest part already exists.

## Provider page messaging rule
The providers page should feel collaborative, not extractive.
The message is:
- better discovery
- better structure
- better representation
- low-friction participation path later
