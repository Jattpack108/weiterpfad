# 06 — FEATURE SYSTEM AND MATCHING

## First-release features that must feel real
- category navigation
- visible matching or filtering demo
- provider cards / listing cards
- structured provider profile concept
- roadmap or feature maturity explanation
- credible contact / early access path

## Why the matching demo matters
The product promise is not “we also have listings”.
The product promise is “we help reduce the field”.
The demo makes that visible.

## Matching logic principles
Start simple and understandable.

### Inputs can include
- offer type
- support / need tags
- transition relevance
- practical considerations like distance
- availability signal later

### Ranking principle
First:
- category fit
- need fit

Then:
- profile relevance
- distance / practical logic

## Card logic
A provider or offer card should be able to show:
- title
- provider
- city / region
- distance
- category
- support tags
- short description
- update state
- optional practical note
- later availability state

## Why comparison matters later
Families do not only need results.
They need to compare realistic alternatives.
The site can mention this future direction without faking the full feature.

## Features to prepare for but not fake now
- compare view
- saved shortlist
- enquiry flow
- provider verification dashboard
- availability tracking
- waitlist signals
- parent checklist / status tracker

## Rule for the first version
Make the structure visible now.
Do not simulate a fake operating platform.
